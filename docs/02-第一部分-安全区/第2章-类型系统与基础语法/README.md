# 第2章 类型系统与基础语法：`let` 的多种化身

## 本章导读

训练你在一行 C++ 声明里快速拆出可变性、引用关系与真实类型。

## 适合谁读

需要把 Rust 的绑定直觉迁移到 C++ 声明阅读中的读者。

Rust 开发者阅读 C++ 基础语法时，最容易产生一种错觉：变量声明、引用、`const`、类型推导看起来都能找到对应物，所以应该不难。难点恰恰在这里——**对应物存在，但边界和默认语义不同。**

本章的重点不是穷举语法，而是帮助你在看到一行声明时，快速判断：这个对象是否可变、是否拥有数据、是否可能悬垂、类型到底是谁。

## 2.1 不可变绑定：`let` 与 `const`；可变绑定：`let mut` 与非 `const`

Rust 中，绑定默认不可变：

```rust
let x = 1;
let mut y = 2;
```

C++ 中，局部变量默认可变：

```cpp
int x = 1;
const int y = 2;
```

### 阅读时的第一个反射动作

看到 C++ 声明时，不要假设“默认不可变”，而是反过来：

- 没写 `const`，通常就是可改的
- 写了 `const`，还要进一步看它修饰的是对象本身、指针、指向对象，还是成员函数上下文

## 2.2 类型推导：`auto`、`decltype(auto)` 与 Rust 的自动推导

Rust 的类型推导总体较稳定，且受所有权与借用规则约束。C++ 的 `auto` 很常见，但推导结果容易被引用、`const`、值类别影响。

```cpp
auto x = expr;
auto& r = expr;
const auto& cr = expr;
```

### 阅读线索

- `auto x`：通常按值接收，可能发生拷贝或移动
- `auto& x`：绑定左值引用，不拥有对象
- `const auto& x`：常用于只读借用，避免拷贝
- `auto&& x`：在模板与泛型上下文里语义复杂，不能简单理解成 Rust 的可变借用

`decltype(auto)` 则更偏高级阅读场景，它保留表达式原始类型形式，尤其会保留引用性。看到它时，应提高警惕，确认返回或绑定的是值还是引用。

## 2.3 引用与指针：`&T` / `&mut T` 到 `T&` / `T*` 的映射

Rust 中，引用天然带借用语义，且生命周期受编译器约束。C++ 中虽然也有引用与指针，但它们只表达“访问方式”，不自动附带完整的安全保证。

### 一个粗略映射

| Rust | C++ | 备注 |
| --- | --- | --- |
| `&T` | `const T&` | 常见近似对应，但不等价 |
| `&mut T` | `T&` | 缺少 Rust 独占借用保证 |
| `*const T` / `*mut T` | `T*` / `const T*` | 原始指针语义更弱 |

### 阅读时要额外补问的问题

- 这个引用是否会悬垂？
- 这个指针是否允许为空？
- 是否存在别名修改？
- 被引用对象的生命周期由谁保证？

Rust 中这些问题很多由编译器先回答；C++ 中往往要靠你自己追踪。

## 2.4 `const` 的多重身份：顶层 const、底层 const、成员函数 const

这是 Rust 开发者读 C++ 时非常容易混淆的点。

### 顶层 const

```cpp
const int x = 42;
```

表示 `x` 本身不可改。

### 底层 const

```cpp
const int* p;
int const* p;
```

表示“指向的对象不可通过 `p` 修改”。

### 指针自身 const

```cpp
int* const p = &x;
```

表示 `p` 这个指针值本身不能改指向，但可通过它修改目标对象。

### 成员函数 const

```cpp
class User {
public:
    int id() const;
};
```

表示该成员函数承诺不修改对象的逻辑状态，至少不能修改普通成员。

### 阅读建议

看到复杂声明时，不要强行心算。先拆成自然语言：

- 这个变量本身能不能改？
- 它指向的数据能不能通过它改？
- 这个方法是在只读对象上也能调用吗？

## 2.5 内建类型与固定宽度整数（`int32_t`、`size_t` 等）

Rust 的整数类型更显式，如 `u32`、`i64`、`usize`。C++ 中既有传统内建类型，也有固定宽度类型。

### 阅读时的基本原则

- `int`、`long`、`long long` 的位宽与平台相关，不要凭名字想当然。
- 需要明确位宽时，优先关注 `int32_t`、`uint64_t` 等固定宽度类型。
- `size_t` 常用于大小、长度、索引，但它是无符号类型，和有符号整数混用时要小心。

## 2.6 控制流：`if`、`switch`、范围 `for` 与 Rust 的对应

C++ 的控制流表面并不陌生，但有若干阅读差异：

- `if` 是语句，不是表达式
- `switch` 更常见，也更容易因 `break`、`fallthrough` 产生误读
- 范围 `for` 很像 Rust 的 `for`，但底层依赖迭代器协议

```cpp
for (const auto& item : items) {
    use(item);
}
```

阅读这类循环时，除了看循环体，还要确认：

- `item` 是值、副本，还是引用
- 容器在循环中是否被修改
- 该修改是否会导致迭代器失效

## 2.7 习语解码：`auto&&` 万能引用初见

`auto&&` 是初学阶段最容易“先背术语、后忘语义”的写法。

在泛型上下文里，它通常意味着一种**保留值类别的转发绑定**。在普通范围 `for` 中，也可能只是“尽量泛化地接住元素”。

```cpp
for (auto&& elem : container) {
    // ...
}
```

阅读时不必先把全部模板规则背熟，但至少要知道：

- 它不是简单的“右值引用”
- 它经常与模板推导、完美转发有关
- 出现在库代码里时，通常意味着作者在控制拷贝与引用折叠行为

## 2.8 练习：标注一段函数的 `const` 与所有权语义

请尝试阅读下面的签名：

```cpp
std::string_view name() const;
void set_name(const std::string& value);
const Config* config() const;
```

为每个函数回答：

1. 返回的是拥有值还是观察视图？
2. 调用者是否需要管理返回对象的生命周期？
3. 参数是按值、只读借用还是其他形式？
4. 成员函数上的 `const` 限定意味着什么？

## 2.9 常见误判

- `const T&` 不等于 Rust 里受借用规则保护的 `&T`。
- `T&` 不等于 Rust 的独占可变借用。
- `auto` 不等于“不会发生拷贝”，它常常就是按值接住结果。
- `decltype(auto)` 不等于“更智能的 auto”，它经常意味着引用性会被保留。
- 成员函数后的 `const` 不等于对象绝对不可变，它主要约束通过该方法修改普通成员。

## 2.10 阅读时先问什么

看到一行声明时，先问：

1. 这是值、引用还是指针？
2. 这个名字本身可不可改，所指向的数据可不可改？
3. 这里是否发生了按值接收、引用绑定或潜在拷贝？
4. 返回值或成员里有没有观察视图而不是拥有值？

## 本章检查清单

- 看到声明时，能先判断值、引用、指针三种访问方式
- 知道 C++ 局部变量默认可变
- 遇到 `auto` 时会主动确认是否发生拷贝或绑定引用
- 能拆解 `const` 是修饰对象本身、指针还是成员函数上下文
- 不会把 C++ 引用直接等同于 Rust 借用
- 看到复杂声明时会先翻译成自然语言再判断语义
