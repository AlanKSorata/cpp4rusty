第3章 智能指针：所有权在 C++ 中的投影

本章导读

把 C++ 资源所有权重新拉回可推理状态。

适合谁读

想先抓住“谁拥有谁”这一核心线索再继续深入 C++ 的读者。

对 Rust 开发者来说，理解 C++ 智能指针往往是建立阅读信心的关键一步。因为在这一部分，你终于能重新看到“谁拥有资源”“谁负责释放”这类熟悉问题的明确线索。

但要注意，C++ 智能指针不是 Rust 所有权系统的等价移植。它们是库类型，不是语言级借用规则；它们表达的是资源管理策略，而不是完整的别名与生命周期约束。

3.1 Box<T> ↔ std::unique_ptr<T>：独占所有权

std::unique_ptr<T> 是最接近 Rust Box<T> 阅读直觉的类型。

std::unique_ptr<Node> node = std::make_unique<Node>();

你可以先抓住的核心点

  • 同一时刻通常只有一个拥有者
  • 对象会在 unique_ptr 销毁时自动释放
  • 不能随意拷贝，只能移动

阅读时的意义

看到 unique_ptr，你通常可以先假设：

  • 这里有明确的资源归属
  • 代码倾向于 RAII 管理
  • 所有权转移会通过移动语义显式发生

这是 C++ 代码里比较“让人安心”的信号。

3.2 Rc<T> / Arc<T> ↔ std::shared_ptr<T>：共享所有权与控制块

std::shared_ptr<T> 表示共享所有权，底层通常配合引用计数控制块。

auto user = std::make_shared<User>();

与 Rust 直觉的相似点

  • 多个对象可以共同拥有同一资源
  • 最后一个拥有者离开时释放资源

需要特别注意的不同点

  • shared_ptr 本身不自动解决可变性协调问题
  • 多个 shared_ptr 指向同一对象，并不意味着访问是线程安全的
  • 控制块位置、构造方式、别名构造等细节可能影响性能与语义

阅读时，一旦看到 shared_ptr，就应立刻多问一句：

为什么这里需要共享所有权，而不是更简单的唯一所有权？

因为共享所有权通常意味着对象边界更模糊，生命周期更难推断。

3.3 Weak<T> ↔ std::weak_ptr<T>：打破循环

std::weak_ptr<T> 是对 shared_ptr 所管理对象的非拥有观察者。

它解决的问题

如果两个对象都用 shared_ptr 互相持有，就可能形成循环引用，导致引用计数永远无法归零。

std::weak_ptr<Node> parent;
std::shared_ptr<Node> child;

阅读重点

  • weak_ptr 不延长对象生命周期
  • 使用前通常要 lock() 成 shared_ptr
  • lock() 失败意味着对象已被释放

看到 weak_ptr 时，你可以把它当作“这是一条观察边，而不是拥有边”的明确信号。

3.4 所有权转移：std::move 的真相与移动后状态

这是 Rust 开发者最容易“半懂半误会”的点。

std::move 本身不移动任何东西，它只是把表达式转换成一个可被移动的值类别，真正的移动发生在移动构造或移动赋值中。

auto p = std::make_unique<Foo>();
auto q = std::move(p);

阅读时你应得出的结论

  • p 的所有权被转交给 q
  • p 进入“有效但未指定值”或空状态，具体取决于类型
  • 对 unique_ptr 而言，移动后通常变成 nullptr

不要把 std::move 直接想成 Rust 的 move 语义关键字；它更像一种“允许后续执行移动操作”的显式标记。

3.5 惯用创建：std::make_unique / std::make_shared

现代 C++ 里，通常优先使用：

  • std::make_unique<T>(...)
  • std::make_shared<T>(...)

而不是手写 new。

为什么这值得你在阅读时关注

因为这通常说明代码在遵循现代资源管理风格，减少了裸 new 带来的异常安全和可读性问题。

auto conn = std::make_unique<Connection>(cfg);

看到这类写法时，你往往能更容易判断对象的创建点与拥有者。

3.6 裸指针的获取：.get() 与非拥有观察者

智能指针并不意味着世界里不再有裸指针。很多 API 会通过 .get() 暴露底层原始指针。

Foo* ptr = foo_ptr.get();

阅读时不要误判

.get() 不会转移所有权。它只是借你一个观察入口。

因此你要追问：

  • 这个裸指针只是临时使用，还是被长期保存？
  • 保存方是否误以为自己拥有它？
  • 原始拥有者在这段期间会不会释放对象？

3.7 习语解码：= delete 与禁止拷贝

你经常会在资源类里看到：

class File {
public:
    File(const File&) = delete;
    File& operator=(const File&) = delete;
};

这表示显式禁止拷贝。对 Rust 开发者来说，可以把它理解为“该类型不实现 Clone 风格的复制语义，并且作者明确防止误拷贝资源”。

这通常是好信号：说明作者意识到了资源唯一性。

3.8 陷阱：循环引用泄漏、误用 new 赋值智能指针

循环引用

如果两个对象互持 shared_ptr，即使外部不再引用它们，也不会自动释放。

误用裸 new

std::shared_ptr<Foo> p(new Foo());

这不是一定错误，但通常不如 make_shared 清晰，也更容易在复杂表达式中埋下异常安全问题。

阅读时，如果一个仓库大量混用智能指针和手写 new / delete，就要提高警惕。
