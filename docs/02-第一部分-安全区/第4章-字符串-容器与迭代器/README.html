第4章 字符串、容器与迭代器：集合生态的对比

本章导读

帮助你识别最常见集合类型背后的生命周期与失效风险。

适合谁读

已经认识 STL 名字，但对 string_view、迭代器和容器失效没有稳定直觉的读者。

Rust 开发者在阅读 C++ 容器代码时，往往能迅速找到熟悉对象：std::string 像 String，std::vector<T> 像 Vec<T>，std::map / std::unordered_map 也不难理解。真正的阅读难点不在“它们是什么”，而在“它们什么时候失效、什么时候悬垂、什么时候隐式拷贝”。

这一章的核心目标，是帮你把容器阅读从“API 名称识别”提升到“内存与生命周期风险识别”。

4.1 String ↔ std::string，&str ↔ std::string_view（C++17）

std::string

std::string 是拥有型字符串容器，大体可类比 Rust 的 String。

std::string name = "alice";

std::string_view

std::string_view 更接近一种“只读字符串切片视图”。

std::string_view view = name;

关键差异

Rust 的 &str 受借用系统约束；C++ 的 string_view 不会自动帮你追踪底层数据是否还活着。

因此阅读时一定要问：

  • string_view 指向的是长期存在的缓冲区，还是临时对象？
  • 原字符串是否可能在视图使用期间被销毁或修改？

4.2 Vec<T> ↔ std::vector<T>：内存布局与增长策略

std::vector<T> 是最常见、最值得优先熟悉的容器。

std::vector<int> nums = {1, 2, 3};
nums.push_back(4);

Rust 开发者容易先想到的点

  • 连续内存
  • 动态扩容
  • 下标访问

阅读时还应额外想到的点

  • 扩容可能导致元素搬迁
  • 指向元素的指针、引用、迭代器可能失效
  • push_back、emplace_back、insert 可能改变先前拿到的地址稳定性

这正是 C++ 阅读与 Rust 阅读的重要差别之一：你需要主动追踪“地址是否还有效”。

4.3 哈希与有序：std::unordered_map / std::map 与 Rust 对应物

可以粗略类比：

  • std::unordered_map<K, V> ↔ HashMap<K, V>
  • std::map<K, V> ↔ BTreeMap<K, V>

阅读时的重点不是 API，而是语义

  • unordered_map 以哈希表为基础，关注平均 O(1) 查找
  • map 以有序树结构为基础，键有序，关注稳定顺序与范围操作

当项目选择其中一种容器时，通常反映了作者对顺序、性能、稳定性或迭代行为的需求。

4.4 迭代器体系：begin() / end() 与范围 for 的语法糖

C++ 的容器遍历大量建立在迭代器之上。范围 for 只是语法糖。

for (auto it = v.begin(); it != v.end(); ++it) {
    use(*it);
}

for (const auto& item : v) {
    use(item);
}

阅读建议

看到循环时，先判断：

  • 当前拿的是元素副本、引用，还是迭代器本身
  • 容器是否在循环体内被修改
  • 修改是否会使当前或后续迭代器失效

4.5 迭代器失效：何时安全？何时悬垂？

这是容器阅读中的高频风险点。

例如对 vector 而言：

  • 扩容可能使所有元素地址变化
  • 插入和删除可能使部分或全部迭代器失效

对 Rust 开发者来说，可以把它理解为一种编译器不会替你兜底的借用失效问题。

阅读时的危险信号

  • 在遍历 vector 时执行 push_back
  • 保存 data()、元素引用、元素指针后又修改容器
  • 删除当前元素后继续使用旧迭代器

4.6 习语解码：emplace_back、reserve、shrink_to_fit

emplace_back

用于原地构造元素，避免某些中间对象。

reserve

提前预留容量，减少扩容次数，也常用于降低迭代器失效风险。

shrink_to_fit

请求容器回收多余容量，但不保证一定执行。

阅读时，这些调用不只是“性能优化细节”，还可能透露作者对对象搬迁与地址稳定性的关注。

4.7 常见漏洞：string_view 的悬挂、越界访问

string_view 悬挂

std::string_view f() {
    return std::string("tmp");
}

这会返回指向已销毁临时对象的视图。

越界访问

int x = v[10];

operator[] 通常不做边界检查。和 Rust 的安全索引习惯不同，这里很可能直接走向 UB。

4.8 练习：阅读并修复一段有迭代器失效问题的循环

请分析下面的代码：

for (auto it = items.begin(); it != items.end(); ++it) {
    if (need_more(*it)) {
        items.push_back(make_item());
    }
}

思考：

  1. 哪一步可能让 it 失效？
  2. 失效后继续 ++it 会发生什么？
  3. 如果这是 Rust，编译器大概率会阻止哪种别名/修改模式？

4.9 常见误判

  • string_view 不等于安全版 &str。
  • vector 不等于“元素地址天然稳定”的动态数组。
  • 范围 for 不等于脱离迭代器模型，它只是语法糖。
  • reserve() 不等于以后永远不会失效，它只是减少失效风险。
  • operator[] 不等于 Rust 那种默认受保护的安全索引。

4.10 阅读时先问什么

看到容器和字符串代码时，先问：

  1. 这里拿到的是拥有对象、元素引用，还是观察视图？
  2. 容器在这段期间会不会扩容、删除或重排？
  3. 之前保存的指针、引用、迭代器会不会因此失效？
  4. 这里是否在用不检查边界的访问方式？
