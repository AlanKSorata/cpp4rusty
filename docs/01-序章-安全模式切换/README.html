序章：安全模式切换——从 Rust 到 C++ 的心智准备

本章导读

从 Rust 转向 C++ 阅读时，最重要的不是先记住语法，而是先切换心智模型。你需要接受一个事实：很多在 Rust 中由语言和编译器自动保障的性质，在 C++ 中只是约定、习惯，或者根本没有保障。

0.1 哲学对照：零成本抽象、信任程序员 vs 编译期保障

Rust 和 C++ 都强调性能与抽象能力，但默认立场不同：

维度RustC++
默认安全性安全默认开启，unsafe 显式标记语言大部分能力默认可用
所有权约束编译器强制检查主要依赖约定与编码风格
生命周期表达类型系统直接参与通常需要读代码推断
错误暴露方式提前在编译期暴露很多问题许多问题可延迟到运行时甚至成为 UB

对 Rust 开发者来说，这意味着你在读 C++ 时必须主动补上缺失的显式边界。

0.2 本书的阅读策略：“先安全，后边界，再激战”

全书按三个阅读层次展开：

  1. 先安全：先看那些相对稳定、可映射到 Rust 的内容，例如项目结构、基础类型、智能指针、容器。
  2. 后边界：再看那些表面相似但语义不同的机制，例如类层次、异常、variant、对象生命周期。
  3. 再激战：最后处理模板、宏、未定义行为、大型项目阅读路径等高复杂度主题。

这个顺序的目的，是让你始终先建立可靠的锚点，再逐步进入不稳定地带。

0.3 理解你在读“全局 unsafe 代码”这一核心前提

一个很实用的阅读方法是：把 C++ 当作一门默认运行在全局 unsafe 上下文中的语言。

这并不意味着所有 C++ 代码都糟糕，而是意味着：

  • 指针、引用、迭代器、对象生命周期都需要你主动验证。
  • “这看起来应该没问题”不是证据。
  • API 签名通常不足以完整表达借用关系和生命周期约束。
  • 代码能编译，不等于它是内存安全的，也不等于它没有 UB。

所以在阅读时，先问：

  • 资源由谁释放？
  • 这段引用是否可能悬空？
  • 这里是否依赖调用顺序或隐含前提？
  • 这个容器操作会不会让先前拿到的指针/迭代器失效？

0.4 代码示例中的对照范式说明

本书中的大多数讲解遵循类似结构：

  1. 给出 Rust 中的熟悉概念作为参照。
  2. 展示 C++ 中最接近的语言或标准库对应物。
  3. 说明“看起来相似”的地方。
  4. 指出“真正危险”的差异。
  5. 总结阅读时应重点检查的线索。

例如：

  • Box<T> ↔ std::unique_ptr<T>
  • Option<T> ↔ std::optional<T>
  • trait object ↔ 虚函数 + 基类指针/引用
  • Drop ↔ RAII + 析构函数

但这些都不是一一等价替换，而只是阅读入口。

本章检查清单

进入后续章节前，请先确认你已经接受以下前提：

  • 不把 C++ 的“能编译”当作“安全”的证明。
  • 不把 Rust 的直觉直接套到 C++ 上。
  • 阅读时优先检查所有权、生命周期、异常路径、容器失效、继承层次与宏边界。
  • 允许自己先建立“足够正确”的局部理解，再逐步补全细节。
