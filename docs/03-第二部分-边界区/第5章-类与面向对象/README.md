# 第5章 类与面向对象：`struct` + `impl` 之外的世界

## 本章导读

把继承、多态和对象模型从“看着像旧语法”变成“可追踪的运行结构”。

## 适合谁读

不常写面向对象代码，但必须读懂 C++ 类层次与虚函数分发的读者。

Rust 开发者读 C++ 面向对象代码时，最容易出现两种极端：要么觉得“这不就是 `struct` 加方法吗”，要么觉得“继承体系太老派，不值得理解”。这两种看法都不够准确。

在很多 C++ 工程里，类不仅是数据与行为的载体，还是 ABI 边界、资源管理单元、动态分发入口和框架扩展点。你未必喜欢它，但你必须会读它。

## 5.1 定义与访问控制：`class`（private 默认）vs `struct`（public 默认）

C++ 中 `class` 和 `struct` 在能力上几乎一样，最核心差异是默认访问控制：

- `struct` 默认 `public`
- `class` 默认 `private`

```cpp
struct Point {
    int x;
    int y;
};

class User {
    int id_;
public:
    int id() const;
};
```

### 阅读提示

不要把 `struct` 直接等同于 Rust 的 `struct`。在 C++ 工程里：

- 有些团队用 `struct` 表示纯数据
- 有些团队只是代码风格偏好
- 仍然可能存在复杂构造、成员函数和不变量

## 5.2 构造函数/析构函数：RAII 的基石

C++ 类最重要的阅读入口之一，是它的构造与析构逻辑。

```cpp
class File {
public:
    File(const std::string& path);
    ~File();
};
```

### Rust 开发者应优先关注什么

- 构造函数是否建立资源句柄、锁、连接、内存
- 析构函数是否释放资源
- 类型是否依赖 RAII 维持不变量

你可以把它理解为“`Drop` 被拆成了更自由但也更危险的一整套对象生命周期机制”。

## 5.3 继承：`class Derived : public Base` 与动态多态

继承是许多 C++ 框架和老代码的核心组织方式。

```cpp
class Base {
public:
    virtual void run();
};

class Derived : public Base {
public:
    void run() override;
};
```

### 阅读重点

- `public` 继承常表示 is-a 关系
- 基类接口常用于多态调用
- 派生类常承担具体实现

对 Rust 开发者来说，近似理解路径通常是：

- 基类 ≈ 抽象接口载体
- 虚函数 ≈ 动态分发入口
- 基类指针/引用 ≈ trait object 的部分阅读效果

但这只是近似，C++ 不会替你自动维持对象安全边界。

## 5.4 虚函数表：`virtual` 如何实现 `dyn Trait`

当成员函数被声明为 `virtual`，通过基类指针或引用调用时，实际分派目标会在运行时决定。

### 阅读时应识别的模式

- 基类中声明 `virtual`
- 派生类中使用 `override`
- 某处持有 `Base*`、`Base&`、`std::unique_ptr<Base>` 或 `std::shared_ptr<Base>`

这通常意味着：

- 代码设计成可替换实现
- 真正执行路径要沿动态类型继续追踪
- 静态阅读时不能只看基类实现

## 5.5 纯虚函数与抽象基类：trait 的对应物

```cpp
class Shape {
public:
    virtual double area() const = 0;
};
```

纯虚函数使类成为抽象基类，不能直接实例化。它是 C++ 中最接近“接口契约”的传统机制。

### 阅读意义

- 看到 `= 0` 时，知道这里定义的是扩展点
- 需要寻找所有实现类
- 真实逻辑分散在派生层，不在抽象定义处

## 5.6 多重继承与虚基类：菱形问题的根源

这是 Rust 开发者最容易感到陌生、也最容易暂时跳过的区域。

### 你至少应该知道什么

- 一个类可以继承多个基类
- 这可能导致同一祖先子对象出现多份副本
- 虚继承用于解决某些共享基类布局问题

阅读老项目或框架代码时，如果遇到多重继承，不必先完全掌握对象布局细节，但一定要提高警惕：

- 名字解析可能更复杂
- 构造顺序与销毁顺序更难直观判断
- 指针转换和对象模型可能不再简单

## 5.7 对象切片与非虚析构陷阱：从 Rust 视角看为什么危险

### 对象切片

```cpp
Base b = Derived();
```

这会把派生类对象按值截成基类子对象，丢失派生部分。对 Rust 开发者来说，这很像“你以为保留了完整动态对象，实际上只剩一个被裁剪后的静态壳”。

### 非虚析构

如果一个类将被用作多态基类，却没有虚析构函数，通过基类指针删除派生对象就可能出现未定义行为或资源泄漏。

```cpp
class Base {
public:
    ~Base();
};
```

如果这个类还有虚函数，却析构不虚，阅读时要立刻警觉。

## 5.8 习语解码：`override`、`final`、NVI 模式

### `override`

表示“我确实在重写基类虚函数”。这是很好的阅读信号。

### `final`

用于禁止继续重写或继承。

### NVI（Non-Virtual Interface）

一种常见模式：公开的非虚函数内部调用受保护的虚函数。这样可以把通用前后处理放在基类，把变化点留给派生类。

阅读时，如果看到基类公开方法很短、内部转调另一个受保护虚函数，这往往就是 NVI 味道。

## 5.9 练习：重构一个继承层次，用 Rust trait object 重新表达

阅读一个典型层次：

- `Base` 定义接口
- `DerivedA`、`DerivedB` 分别实现
- 代码通过 `std::unique_ptr<Base>` 持有实例

尝试回答：

1. 哪部分近似对应 trait 定义？
2. 哪部分近似对应 `Box<dyn Trait>`？
3. 哪些风险是 Rust trait object 不会默认暴露出来的？

## 5.10 常见误判

- `struct` 不等于 Rust 里的“纯数据结构”。
- 有继承不等于一定在做运行时多态，可能只是代码复用或框架约定。
- 有虚函数不等于真实逻辑就在基类里，很多时候真正逻辑在派生类。
- 基类指针持有对象不等于自动安全，析构和生命周期仍要手查。
- 看起来像 trait object 的结构不等于享有 Rust 那种对象安全与生命周期约束。

## 5.11 阅读时先问什么

进入类层次时，先问：

1. 这个类主要是数据载体、资源句柄，还是多态接口？
2. 构造和析构里做了什么关键动作？
3. 哪些函数是虚函数，真实分派目标在哪里？
4. 是否存在按值传递、多态删除或对象切片风险？

## 本章检查清单

- 区分 `class` 与 `struct` 的默认访问差异
- 读类时优先查看构造、析构、成员和虚函数
- 识别虚函数、多态持有和抽象基类
- 知道对象切片是按值语义带来的真实风险
- 看到多态基类时会检查析构函数是否为虚函数
- 进入类层次时会主动区分“接口角色”和“资源角色”
