第6章 错误处理：异常、Optional 与 Expected

本章导读

建立阅读 C++ 多轨错误处理风格的判断框架。

适合谁读

习惯 Result/Option，但需要进入异常、错误码和混合策略工程的读者。

Rust 开发者在读 C++ 错误处理时，最容易困惑的点是：同一个项目里，可能同时存在异常、错误码、布尔返回、空指针、optional，甚至日志后直接退出。它不像 Rust 那样有一条明显的主航道。

因此，本章的重点不是判断“哪种方式最好”，而是学会在阅读中识别：错误是如何传播的、清理是如何发生的、调用者被要求承担什么责任。

6.1 异常机制：throw / try / catch 与 panic! / 普遍捕获的差异

C++ 异常并不等于 Rust panic!。更接近的理解是：一种可跨函数边界传播的错误通道。

try {
    parse(input);
} catch (const std::exception& e) {
    log_error(e.what());
}

阅读时的重点

  • 哪些函数可能抛异常？
  • 异常在哪里被捕获？
  • 中间层是否只是透传？
  • 资源释放是否依赖析构自动完成？

Rust 中你往往沿 Result 链追踪；C++ 中你需要同时留意“正常返回路径”和“抛出路径”。

6.2 noexcept 承诺：函数契约与潜在谎言

noexcept 表示函数承诺不抛异常。

void close() noexcept;

阅读意义

  • 这是 API 契约的一部分
  • 标准库和容器优化会利用该信息
  • 如果 noexcept 函数内部真的抛出异常，通常会导致程序终止

因此，看到 noexcept 时不要只把它当装饰词，而要意识到它改变了错误传播模型。

6.3 std::optional<T>：C++17 的 Option<T>

std::optional<T> 是比较接近 Rust Option<T> 的标准类型。

std::optional<int> parse_int(std::string_view s);

阅读提示

  • 它表达“可能没有值”
  • 但通常不携带失败原因
  • 适合缺失是常规分支、错误信息不重要的场景

如果你看到一个解析函数返回 optional，就要自然补问：

无值到底表示输入不合法、资源不存在，还是某种条件未满足？

6.4 std::expected<T, E>（C++23）与错误码流派

std::expected<T, E> 更接近 Rust Result<T, E> 的阅读体验。

std::expected<Value, ParseError> parse(std::string_view input);

但要注意

  • 很多老项目没有 C++23，仍使用自定义 Expected、Outcome 或错误码模式
  • 即使有 expected，生态也不像 Rust 那样统一

阅读时不要只认标准名，还要识别“项目自制的 Result 风格容器”。

6.5 工程中的混合错误策略：何时异常？何时返回？

现实项目经常混用多种策略：

  • 构造失败用异常
  • 可预期缺失用 optional
  • 业务失败用错误码或 expected
  • 低层 C API 用返回值加输出参数

阅读建议

不要急着评判风格，先画出边界：

  • 哪一层偏向异常
  • 哪一层偏向返回值
  • 哪些边界必须把异常翻译成错误码，或反之

这比单看单个函数更能解释整个项目的错误处理形状。

常见分层图

很多项目的真实形状更接近：

  • 应用层：允许异常，集中在入口统一捕获
  • 库边界层：把异常翻译成 expected、状态对象或错误码
  • C API / ABI 边界：禁止异常继续穿越

一旦你看出这个分层，项目里的“混用”就会变得可解释。

6.6 析构函数与异常：铁律“绝不抛出”

一个重要经验规则是：析构函数不应让异常逃逸。

原因很简单：如果在栈展开期间析构函数再抛异常，程序通常会直接终止。

因此阅读 RAII 类型时，如果析构函数中含有可能失败的动作，你应重点关注：

  • 它是否吞掉错误
  • 是否转成日志
  • 是否提供单独的显式 close() / commit() API 处理失败

Rust 开发者的直觉对照

这和 Rust 里“不把真正可能失败的业务动作塞进 Drop”有相近精神。真正重要的失败路径，通常还是要靠显式 API 处理。

6.7 习语解码：std::error_code 与 boost::outcome 速览

std::error_code

偏向轻量、可组合的错误表示，适合系统调用和库边界。

boost::outcome

在标准 expected 进入语言前，一些项目会用它表达结果与错误。

阅读时，不需要一开始就掌握所有 API，但至少要认出：这些类型是在为“非异常错误通道”提供统一包装。

6.8 异常与 ABI / C 边界：不要让异常穿越不该穿越的地方

工程里一个非常关键的问题是：异常能不能跨边界传播。

你需要先记住

  • C++ 异常不应穿过 C API 边界
  • 动态库边界是否允许异常穿越，取决于 ABI、编译器和项目约定
  • 对外稳定接口常常会把失败翻译成错误码、状态对象或输出参数

例如：

extern "C" int parse_json(const char* input, JsonValue* out_value);

看到这类接口时，阅读重点应转成：

  • 内部异常是否在边界前被捕获
  • 捕获后如何翻译为返回值或错误对象
  • 错误信息是否被记录、丢弃或另行存储

典型信号

  • extern "C"
  • 对外导出函数带 noexcept
  • Status / ErrorCode / Result 风格的公开接口

这些通常都在提示你：这里是错误传播模型的翻译层。

6.9 禁用异常项目：读 -fno-exceptions 风格代码

有些项目会全局禁用异常，或者实际上一律不用异常。

这时你更可能看到

  • bool / int / error_code 返回值
  • 输出参数承载结果
  • 自定义 StatusOr<T> / Expected<T, E>
  • 失败后记录日志并立刻返回

阅读策略

  • 先确认失败是否必须由调用者显式检查
  • 再确认错误对象是否保留足够上下文
  • 最后确认资源清理是否完全依赖 RAII，而不是异常展开

如果项目禁用异常，就不要再沿“谁 catch 了它”这条线读，而应改成沿返回值链和状态对象链追踪。

6.10 练习：将一个可能出现错误的解析函数改写为多种 C++ 错误风格

设想一个字符串解析函数，你可以分别思考以下签名的阅读含义：

Value parse(std::string_view s);
std::optional<Value> parse(std::string_view s);
std::expected<Value, ParseError> parse(std::string_view s);
Value parse_or_throw(std::string_view s);

请分别回答：

  • 调用者如何知道失败？
  • 错误信息是否保留？
  • 失败路径是否与正常路径显式分离？

6.11 阅读时先问什么

进入一个陌生模块时，先问：

  1. 这个模块主要靠异常还是返回值报告失败？
  2. 错误边界在哪里发生翻译？
  3. 析构、回滚、清理动作依赖什么机制？
  4. 对外接口是否禁止异常穿越？

  • 不把异常简单等同于 panic!
  • 知道 optional 适合表达“可能无值”，不擅长表达详细错误
  • 知道 expected 更接近 Result
  • 看到 noexcept 会意识到它是传播契约
  • 阅读项目时会先识别整体错误处理策略，而不是只看单点 API
