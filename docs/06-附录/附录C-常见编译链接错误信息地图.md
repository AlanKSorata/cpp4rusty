# 附录 C：常见编译/链接错误信息地图

## 编译期高频错误

### `unknown type name` / `does not name a type`

常见原因：

- 缺少头文件
- 命名空间前缀错误
- 宏条件导致定义未出现
- 前向声明不足以支撑当前用法

下一步去看哪里：

- 当前文件是否真的 `#include` 了正确头文件
- 命名空间前缀是否写全
- 条件编译是否把定义屏蔽掉了

### `use of incomplete type`

常见原因：

- 只有前向声明，没有完整定义
- 试图按值持有或访问成员

下一步去看哪里：

- 这个类型当前是否只有前向声明
- 是否在头文件中误按值声明了不完整类型成员
- 是否需要把实现移到 `.cpp` 再包含完整定义

### `no matching function for call`

常见原因：

- 参数类型不匹配
- `const` / 引用限定不匹配
- 模板推导失败
- 隐式转换未发生或被 `explicit` 禁止

下一步去看哪里：

- 实参和形参的 `const`、引用、值类别是否匹配
- 是否有重载或模板重载被 `enable_if` / Concepts 筛掉
- 是否误以为会发生某个隐式转换

### 模板相关长错误

阅读建议：

1. 先找第一条真正来自你代码位置的信息。
2. 再找“substitution failed”或约束不满足的位置。
3. 不要一开始就试图从上到下读完整个错误瀑布。

下一步去看哪里：

- 你自己的调用点
- 被实例化的模板签名
- traits / `enable_if` / Concepts 约束

## 链接期高频错误

### `undefined reference` / `unresolved external symbol`

常见原因：

- 只有声明，没有定义
- 定义存在，但未加入 target
- 函数签名不一致
- 模板定义没有放在可见位置
- 链接库缺失

下一步去看哪里：

- 目标源文件是否加入了 `add_library` / `add_executable`
- 头文件声明和实现签名是否完全一致
- 模板定义是否只写在 `.cpp`
- 链接库是否真的出现在 `target_link_libraries`

### 重复定义

常见原因：

- 非内联定义写进头文件并被多个翻译单元包含
- 全局对象或函数定义重复

下一步去看哪里：

- 是否把普通函数定义直接写进了头文件
- 是否漏加 `inline`
- 是否有全局对象在多个源文件重复定义

## Rust 开发者的快速判断法

- **编译错误**：通常是“当前文件看不懂这段代码”。
- **链接错误**：通常是“各文件分别都能编，但最终拼不起来”。

## 阅读建议

遇到错误时，优先定位：

1. 是类型可见性问题，还是符号可见性问题？
2. 是声明和定义断开了，还是 target 和库关系断开了？
3. 是模板实例化不可见，还是普通函数缺失实现？

## 不要误会成什么

- `undefined reference` 不等于“语法写错了”，它更常表示“拼装阶段找不到实现”。
- `use of incomplete type` 不等于“编译器不认识这个类型”，它常表示“只知道名字，不知道完整结构”。
- 模板长错误不等于“整个模板系统都坏了”，通常只是第一处约束没满足。
