第13章 常见 C++ 工程模式识别手册

本章导读

让你通过模式识别而不是逐行硬读来提升陌生项目阅读速度。

适合谁读

进入中大型工程后，需要快速识别单例、工厂、PIMPL、CRTP 等模式的读者。

很多真实项目的难点，不在于单个语言特性，而在于模式识别：你必须迅速判断眼前这一坨代码到底是在实现工厂、单例、PIMPL、静态多态，还是事件回调系统。能识别模式，阅读速度会大幅提升。

13.1 单例（Singleton）：线程安全的 Magic Statics

现代 C++ 中，常见单例写法是函数内静态对象。

Config& global_config() {
    static Config cfg;
    return cfg;
}

阅读时要问

  • 生命周期是否贯穿整个进程
  • 是否引入全局状态耦合
  • 多线程初始化是否依赖语言保证

13.2 工厂：返回 unique_ptr<Base> 的多态创建

std::unique_ptr<Base> make_parser(Type t);

这是非常经典的多态工厂信号。阅读时你通常可以推断：

  • 外部只依赖抽象接口
  • 具体实现由工厂决定
  • 生命周期交给返回的智能指针管理

13.3 PIMPL：编译防火墙与 ABI 稳定

PIMPL 常见结构：

  • 头文件里只暴露类声明
  • 私有成员是 std::unique_ptr<Impl>
  • 真正实现藏在源文件

阅读意义

  • 说明作者在控制编译依赖或 ABI 暴露
  • 看到大量前向声明和 Impl 名字时应优先想到它

13.4 CRTP：静态多态替代 virtual

CRTP 是模板时代的模式识别题：它看起来像继承，但目的通常不是传统运行时多态，而是编译期复用和静态分发。

阅读时如果看到：

template <typename Derived>
class Base {};
class Foo : public Base<Foo> {};

就要意识到这更接近模板技巧而非普通 OO 层次。

13.5 STL 算法替代手写循环：for_each、transform 等

现代 C++ 项目常用算法库表达意图。

阅读时不要只把它看成“写法不同”

  • transform 暗示映射关系
  • all_of / any_of 暗示谓词检查
  • sort + 比较器暗示排序语义核心

算法名往往能比手写循环更快暴露作者意图。

13.6 回调与信号槽：std::function、观察者模式

如果你看到：

  • std::function<void(...)>
  • 回调注册表
  • listener / observer 名字

通常意味着这段代码在做事件驱动或解耦通知。

阅读时要关心：

  • 回调的生命周期由谁保证
  • 是否存在悬挂回调
  • 回调是在同步还是异步上下文触发

13.7 习语解码：在代码中快速识别上述模式

你可以把模式识别压缩成几个“信号词”：

  • static 局部对象 → 单例味道
  • make_* + 返回基类智能指针 → 工厂味道
  • Impl + 前向声明 + unique_ptr → PIMPL 味道
  • Base<Derived> → CRTP 味道
  • std::function / subscribe / emit → 回调系统味道

13.8 练习：在小型开源项目中寻找 3 种模式

请任选一个小型 C++ 项目，尝试找出：

  1. 一个生命周期管理模式
  2. 一个分发/抽象模式
  3. 一个构建或模块解耦模式

13.9 常见误判

  • 单例不等于只是“偷懒的全局变量”，它常常还意味着初始化顺序和生命周期决策。
  • 工厂不等于仅仅“把 new 包一下”，它通常也在隐藏实现选择和所有权策略。
  • PIMPL 不等于无意义的样板，它往往服务于 ABI 或编译隔离。
  • CRTP 不等于普通继承层次，它更多是在做静态多态。
  • std::function 不等于自动安全的回调，生命周期和线程语义仍要手查。

13.10 阅读时先问什么

看到熟悉模式外形时，先问：

  1. 这个模式在这里解决的是生命周期、抽象边界，还是 ABI / 编译隔离问题？
  2. 谁拥有最终对象，谁只持有接口或回调？
  3. 这是运行时多态还是编译期多态？
  4. 这个模式带来的主要成本是什么：全局状态、模板复杂度，还是回调生命周期？
