# 第10章 宏与预处理器：理解旧时代与跨平台的必须

## 本章导读

帮助你越过宏层噪声，直接看见预处理后的真实语义。

## 适合谁读

进入平台代码、老代码或框架代码后频繁被宏阻断阅读节奏的读者。

很多 Rust 开发者一看到 C++ 宏就本能排斥，这种反应可以理解，但对阅读任务没有帮助。宏在现代 C++ 中依然大量存在，尤其是在：

- 平台适配
- 日志与断言
- 头文件配置
- 老代码与大型框架

你不一定要喜欢它，但必须学会在阅读时快速把“宏层噪声”和“真实语义”分开。

## 10.1 对象宏与函数宏：`#define` 的简单替换与危险

```cpp
#define PI 3.14159
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

宏首先是预处理器做的文本替换，而不是类型安全语言构造。

### 阅读时的关键意识

- 它不遵守函数那样的类型规则
- 展开后可能重复求值
- 名字冲突与作用域控制都更弱

## 10.2 条件编译：`#ifdef` / `#if` 与平台、调试分隔

很多项目的大量宏不是为了元编程，而是为了“让不同平台看到不同代码”。

### 阅读时要先区分

- 这是在控制功能开关？
- 还是在控制平台差异？
- 还是在控制编译器兼容？

只有先分清宏的用途，后面读展开结果才不容易迷路。

## 10.3 do-while(0) 与其它包裹技巧

函数式宏常用 `do { ... } while (0)` 包裹，以保证它在语法上更像单条语句。

```cpp
#define LOG(x) do { write_log(x); } while (0)
```

阅读时，这类写法不要逐字纠结它“为什么 while(0)”。把它理解成历史形成的一种安全壳即可。

## 10.4 `#` 与 `##`：字符串化与拼接

- `#x`：把参数转成字符串
- `a##b`：把两个记号拼接

这类写法常见于调试、注册、元信息生成和平台兼容层。

## 10.5 阅读宏密集代码的策略：`-E` 展开、IDE 宏展开功能

如果你感觉肉眼已经无法判断一段代码的真实形态，不要硬读，直接借助工具：

- 预处理展开
- IDE 宏查看
- 跳转宏定义

宏代码最怕的是“凭印象猜”。

## 10.6 常见宏模式：`assert`、日志宏、`FALLTHROUGH`

### `assert`

调试时检查条件，发布构建中可能被关掉。

### 日志宏

常封装文件名、行号、级别和条件编译。

### `FALLTHROUGH`

用于显式标注 `switch` 中有意贯穿到下一分支。

## 10.7 从宏到模板/`constexpr` 的迁移趋势

现代 C++ 倾向于减少宏，把更多逻辑迁移到：

- 模板
- 内联函数
- `constexpr`
- enum class

阅读现代代码时，如果发现作者刻意避免宏，通常是在追求更好的类型安全与调试体验。

## 10.8 练习：展开一个包含嵌套宏的片段并描述其真实语义

面对一段宏嵌套代码，请尝试：

1. 找到最外层宏定义。
2. 展开每一层参数替换。
3. 忽略表面语法奇技淫巧，最后只回答：它最终执行了什么真实语句？

## 10.9 常见误判

- 宏不等于函数调用，它只是预处理期文本替换。
- `#ifdef` 不等于“这段代码当前一定存在”，它恰恰可能说明当前平台根本看不到这段实现。
- 日志宏和断言宏不等于普通业务逻辑，它们常常夹带条件编译和副作用。
- `do { ... } while (0)` 不等于作者在写循环，它只是宏的语法包装。
- 宏很多不等于代码一定落后，可能只是兼容层或框架层特征。

## 10.10 阅读时先问什么

遇到宏密集代码时，先问：

1. 这个宏是在做平台分支、日志包装，还是语法生成？
2. 当前编译条件下，它到底会展开成什么？
3. 宏参数会不会被重复求值？
4. 这层宏背后真正需要读的是哪段普通 C++ 语句？

## 本章检查清单

- 知道宏是文本替换，不是类型安全函数
- 阅读宏时先分辨它是在做平台条件、日志包装还是语法生成
- 遇到复杂宏会主动借助展开工具
- 知道很多现代替代方案已能减少宏使用场景
- 看到 `assert`、日志宏、`FALLTHROUGH` 等模式时不再陌生
- 会把“宏层噪声”和“展开后的真实语义”分开处理
