# 第11章 未定义行为与“隐式 unsafe”心智模型

## 本章导读

建立 Rust 开发者最需要的 C++ 风险审查心智。

## 适合谁读

希望把“这段代码可能出事吗”作为阅读默认问题的读者。

如果说前面几章是在帮 Rust 开发者建立“可读”的 C++ 认知，那么这一章是在建立“可生存”的认知。因为很多真正危险的问题，并不会通过漂亮的 API 名字直接暴露出来，而是藏在默认允许的行为里。

## 11.1 C++ 就是全局 unsafe 代码：Rust 安全边界的反向

一个非常实用的阅读心法是：把 C++ 默认当成全局 `unsafe`。

这不是贬低，而是提醒你：

- 语言不会替你把危险边界标出来
- 你要自己发现可能悬垂、越界、别名冲突、竞争条件的地方

## 11.2 空指针解引用与悬垂引用：经典的 UB 之源

### 空指针

```cpp
Foo* p = nullptr;
p->run();
```

### 悬垂引用

```cpp
const std::string& f() {
    std::string s = "x";
    return s;
}
```

这些都属于阅读时必须立刻标红的模式。

## 11.3 越界与迭代器失效：`operator[]` 对比 `.at()`

- `operator[]` 常不检查边界
- `.at()` 通常执行边界检查并在越界时报告错误

Rust 开发者要特别防止自己把“安全索引默认存在”的直觉带进来。

## 11.4 严格别名规则：类型双关的隐患

某些“把一块内存当另一种不相干类型解释”的操作，可能违反严格别名规则，导致优化器基于错误前提生成代码。

阅读低层性能代码、序列化代码或历史代码时，这是一类很隐蔽的风险。

## 11.5 数据竞争：多线程下的未定义行为

在 Rust 中，数据竞争通常很难悄悄通过类型系统；在 C++ 中，如果多个线程无同步地并发读写同一对象，就可能直接走向未定义行为。

### 阅读时要格外警惕

- 全局状态
- 缓存单例
- 懒初始化
- 引用返回的内部可变对象

## 11.6 生命周期可视化训练：从函数签名绘制对象存活期

阅读复杂函数时，一个很有效的方法是手动画出：

- 哪个对象在栈上
- 哪个对象在堆上
- 谁拥有谁
- 哪个引用/指针指向谁
- 什么时候对象被销毁

这几乎是在手工补上 Rust 编译器平时替你做的部分工作。

## 11.7 C++ 生命周期追踪术：指针/引用来源与拥有者判断

每当你看到指针或引用，优先问：

1. 它来自哪里？
2. 原始拥有者是谁？
3. 拿到它之后，中间是否可能触发对象销毁或搬迁？

这三问足以覆盖大量阅读风险。

## 11.8 卫生工具：AddressSanitizer、UBSan、ThreadSanitizer 简介

这些工具不是语言保障，但它们能在运行时帮助暴露隐藏问题：

- AddressSanitizer：内存越界、use-after-free
- UBSan：部分未定义行为
- ThreadSanitizer：数据竞争

阅读和排查问题时，知道项目是否启用这些工具非常有价值。

## 11.9 高频红旗模式表：看到就该放慢

下面这些模式，在阅读时值得直接标红：

| 模式 | 为什么危险 | 第一反应该查什么 |
| --- | --- | --- |
| 返回局部对象的引用、指针、`string_view` | 生命周期已经结束 | 返回对象的来源与存活期 |
| 容器修改后继续使用旧迭代器/指针 | 扩容或重排可能导致失效 | 修改前后地址是否还有效 |
| 基类无虚析构却通过基类指针释放 | 可能只析构基类部分，甚至 UB | 基类是否声明 `virtual ~Base()` |
| `reinterpret_cast`、手工类型双关 | 可能违反别名规则或对象模型 | 是否只是在读字节，还是在伪装成另一种对象 |
| 无同步共享可变全局状态 | 多线程下可能数据竞争 | 访问路径上是否有锁或原子 |
| 裸 `new` / `delete` 混杂在业务代码里 | 资源职责易混乱 | 是否能追到唯一拥有者 |

这张表的价值不在于立刻下结论，而在于提醒你“这里值得减速”。

## 11.10 Rust 开发者专用的代码审查检查清单

Review C++ 代码时，可优先扫以下问题：

- 是否返回了指向局部对象的引用或视图？
- 容器修改后，旧迭代器/指针是否仍被使用？
- 多态基类是否有虚析构？
- 裸 `new` / `delete` 是否必要且成对？
- 是否存在无同步共享可变状态？
- 异常路径是否破坏资源释放？

## 11.11 低层内存代码的额外问题

遇到序列化、网络、驱动、性能敏感代码时，再额外问：

- 这里是否在手工管理对象布局？
- 是否把字节缓冲区强行解释成复杂类型？
- 是否依赖未写明的对齐、字节序或平台假设？

这类代码不是“普通业务代码稍微危险一点”，而是另一种阅读模式。

## 11.12 常用断言与防御性编程标记

常见信号包括：

- `assert(...)`
- `CHECK(...)`
- `DCHECK(...)`
- `Expects(...)`

它们不能替代类型安全，但能告诉你作者认为哪些前提是关键不变量。

## 11.13 练习：审查一段代码中的潜在 UB，并标注不安全点

请尝试对任意一段陌生 C++ 代码做标注：

- 哪些地方持有裸指针？
- 哪些地方依赖调用顺序？
- 哪些地方可能悬垂或越界？
- 哪些地方在多线程下可能出问题？

## 11.14 阅读时先问什么

一旦进入可疑代码段，先问：

1. 这里谁拥有对象，谁只是观察？
2. 当前指针或引用有没有明确来源？
3. 中间是否可能发生销毁、扩容、移动或线程切换？
4. 这段代码是否依赖平台、ABI 或优化器假设？
